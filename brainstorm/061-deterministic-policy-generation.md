# Brainstorm: Deterministic Policy Generation

**Date:** 2026-05-22
**Status:** active

## Problem Framing

OpenShell network policies are generated by Claude Code interpreting a markdown spec (`cc-deck.build.md` section C4). This is non-deterministic: different runs can produce different policies depending on model reasoning, and getting the policy right requires encoding increasingly complex rules in prose (binary globs, deprecated fields, endpoint grouping, `access` requirements). Every new lesson from testing has to be added to the skill spec and hoped that Claude follows it.

Containerfile snippets solved this problem for image layers: Go templates render deterministic output from structured data. The policy should follow the same pattern.

## Current flow

```
cc-deck.capture --> build.yaml (manifest)
                         |
                         v
cc-deck.build (Claude) ---> reads manifest
                         |   reasons about policy
                         |   generates YAML
                         v
                    openshell/policy.yaml (non-deterministic)
                         |
                         v
                    COPY into image
```

## Proposed flow

```
cc-deck.capture --> build.yaml (manifest)
                         |
                         v
Go GeneratePolicy() ---> reads manifest
                         |   deterministic code
                         |   renders YAML
                         v
                    openshell/policy.yaml (deterministic)
                         |
                         v
cc-deck.build (Claude) just COPYs, never generates
```

## Policy component files

No hardcoded URLs in Go code. Endpoints, binaries, and match rules live in declarative YAML component files. The Go code loads them, matches against the manifest, and assembles the policy.

### Component file format

Each file declares one policy component:

```yaml
name: claude-code
match:
  always: true
endpoints:
  - host: api.anthropic.com
    port: 443
    protocol: rest
    access: full
  - host: statsig.anthropic.com
    port: 443
  - host: downloads.claude.ai
    port: 443
  - host: raw.githubusercontent.com
    port: 443
binaries:
  - /usr/local/bin/claude
  - /sandbox/.local/bin/claude
  - /sandbox/.local/share/claude/**
  - /usr/bin/node
```

### Match semantics

| Field | Meaning |
|-------|---------|
| `always: true` | Always included (claude-code, git-hosting) |
| `tools: [cargo, rust]` | Included when any listed tool appears in manifest |
| `credentials: [vertex, claude-vertex]` | Included when any listed credential type appears |
| `features: [github]` | Included when a GitHub credential or tool is present |

### Component catalog (two tiers)

**Tier 1: Remote catalog (preferred)**

A GitHub repo (`cc-deck/openshell-policies` or similar) serves as a live registry:

```
catalog.yaml              # index: component name -> match rules + version
components/
  claude-code.yaml
  vertex-ai.yaml
  rust.yaml
  golang.yaml
  nodejs.yaml
  python.yaml
  github.yaml
  github-rest.yaml
  ...
```

During `cc-deck.capture`, after detecting tools and credentials:

1. Fetch `catalog.yaml` from the catalog repo (small file, single HTTP request)
2. Match detected components against catalog entries
3. Fetch only the matching component YAML files
4. Cache them locally (`.cc-deck/setup/openshell/components/`)

The catalog is always checked first when network is available, as it may contain fixes and new components that the embedded fallback does not have yet. The catalog repo is versioned via git tags so cc-deck can pin a catalog version if needed.

**Tier 2: Embedded fallback (offline/air-gapped)**

The same component files are embedded in the cc-deck binary via `go:embed`:

```
internal/build/policies/
  claude-code.yaml
  vertex-ai.yaml
  rust.yaml
  ...
```

Used when the catalog is unreachable (offline, air-gapped, corporate firewall). The embedded set is a snapshot from the last cc-deck release. Catalog components override embedded ones by name when both exist.

**Resolution order:**

1. Remote catalog (if reachable)
2. Embedded fallback (if catalog unreachable)
3. User-local overrides (`.cc-deck/setup/openshell/policies/`) always merged on top

### User-local overrides

Users can drop custom component files into `.cc-deck/setup/openshell/policies/` for project-specific needs (MCP servers, internal APIs). These follow the same format and are merged after catalog/embedded components.

### Trust and supply chain

The catalog repo is owned by the cc-deck project. Only the official repo URL is trusted by default. A `--catalog-url` flag allows enterprise forks to point to internal mirrors.

Future consideration: supply chain signing for catalog components (cosign or similar). Deferred to a later spec.

### What the Go code does

The `GeneratePolicy()` function becomes a component assembler:

1. Load embedded component files
2. If catalog is cached locally, overlay catalog components (by name)
3. If user-local overrides exist, overlay those
4. For each component, evaluate `match` against manifest
5. For matching components, merge their endpoints and binaries into the policy
6. Apply `targets.openshell.policy` overrides from manifest (existing merge semantics)
7. Write the final policy YAML

### Components to ship initially

**Always present:**
- `claude-code`: api.anthropic.com (rest, access: full), statsig, sentry, downloads.claude.ai, raw.githubusercontent.com
- `git-hosting`: github.com, gitlab.com with git binary

**Conditional on credentials:**
- `vertex-ai`: bare aiplatform.googleapis.com, regional endpoints, oauth2, www.googleapis.com, accounts.google.com

**Conditional on tools:**
- `rust`: crates.io, static.crates.io, index.crates.io
- `golang`: proxy.golang.org, sum.golang.org, storage.googleapis.com
- `nodejs`: registry.npmjs.org
- `python`: pypi.org, files.pythonhosted.org

**Conditional on features:**
- `github-rest`: api.github.com (rest, access: full), objects.githubusercontent.com
- `github-ssh-over-https`: github.com with git-specific rules

### Binary mapping

Every component file declares its own binaries. Common sets:

| Binary set | Paths |
|------------|-------|
| Claude Code | `/usr/local/bin/claude`, `/sandbox/.local/bin/claude`, `/sandbox/.local/share/claude/**`, `/usr/bin/node` |
| Git | `/usr/bin/git` |
| GitHub CLI | `/usr/bin/gh` |
| Curl | `/usr/bin/curl` |
| Cargo/Rustup | `/sandbox/.cargo/bin/cargo`, `/sandbox/.cargo/bin/rustup` |
| Go | `/usr/local/go/bin/go` |
| Node/npm | `/usr/bin/node`, `/usr/bin/npm`, `/usr/bin/npx` |
| Python/pip | `/sandbox/.venv/bin/python`, `/sandbox/.venv/bin/pip`, `/usr/local/bin/uv` |

## Integration with `build refresh`

`cc-deck build refresh` currently re-extracts snippets and commands. It should also:

1. Read `build.yaml`
2. Load components (cached catalog or embedded fallback + user-local overrides)
3. Match components against manifest
4. Call `GeneratePolicy(manifest, matchedComponents)`
5. Write `openshell/policy.yaml`

This makes `build refresh` the single command for "bring all generated artifacts up to date with the current binary and manifest."

## What stays in the skill

The `/cc-deck.build` command still:
- Reads the manifest for image name, tag, base image
- Assembles the Containerfile from snippets + generated layers (tool installs, settings copies)
- Runs `podman build`
- Handles build failures and retries

It no longer generates the policy. It just COPYs `openshell/policy.yaml` which was pre-rendered by `build refresh` or `build init`.

## What changes in capture

Capture gains a new step after detecting tools, credentials, and MCP servers:

1. Fetch `catalog.yaml` from the remote catalog repo (if reachable)
2. Match detected components against catalog entries
3. Download matching component files
4. Cache them in `.cc-deck/setup/openshell/components/`

This happens at capture time (not refresh time) because capture is when the user is online and interacting. The cached components are then available for `build refresh` to assemble the policy offline.

If the catalog is unreachable, capture logs a warning and moves on. The embedded fallback will be used at refresh time.

## Migration

Existing setups have a Claude-generated `openshell/policy.yaml` with a "DO NOT EDIT" header. After this change:
- `build refresh` overwrites it with the Go-generated version
- Users who manually edited the policy can use `targets.openshell.policy` in the manifest for overrides (merge semantics already implemented)
- The "DO NOT EDIT" header remains accurate since the file is now machine-generated from Go code

## Decisions

- **No hardcoded URLs in Go code.** All endpoints and binaries live in declarative YAML component files, either embedded or from the catalog.
- **Catalog first, embedded fallback.** Always try the remote catalog (it may have fixes), fall back to embedded when offline.
- **Capture fetches, refresh assembles.** Capture downloads catalog components when online. Refresh assembles the policy from cached/embedded components. Clean separation.
- **Supply chain signing deferred.** The catalog repo is trusted by ownership. Cosign or similar signing for catalog components is a future enhancement.

## Open questions

- Should `build refresh` warn if `openshell/policy.yaml` has been manually edited (check for modifications vs. last generated hash)?
- Should MCP server endpoint extraction parse the MCP config format, or should capture store the endpoints explicitly in the manifest? Currently capture stores MCP server names/URLs but not the endpoint host:port pairs.
- What is the right GitHub repo name for the catalog? `cc-deck/openshell-policies`, `cc-deck/policy-catalog`, or something else?
- Should the catalog support component dependencies (e.g., `vertex-ai` depends on `claude-code` binaries)? Or keep it flat with each component self-contained?
