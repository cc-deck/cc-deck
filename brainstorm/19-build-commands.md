# 19: AI-Driven Build Commands

## Problem

Creating a Containerfile that captures all tool dependencies for a set of repositories
requires analyzing diverse build systems (go.mod, package.json, Cargo.toml, pyproject.toml,
Makefile, CI workflows, etc.). This analysis is inherently fuzzy and benefits from LLM
reasoning. Similarly, discovering plugins and MCP servers benefits from conversational
interaction.

## Design: CLI-AI-CLI Flow

The overall workflow follows a three-phase pattern:

```
Phase 1 (CLI):     cc-deck build init <dir>     → scaffold
Phase 2 (AI):      /cc-deck.extract, /cc-deck.plugin, /cc-deck.mcp, /cc-deck.containerfile
Phase 3 (CLI):     cc-deck build, cc-deck push   → deterministic build & publish
```

The user enters Claude Code for Phase 2, uses the slash commands, then exits back to the
CLI for the deterministic steps. This minimizes token usage (build/push don't need AI).

## Commands Detail

### `/cc-deck.extract` - Repository Analysis

**Purpose**: Analyze one or more locally checked-out repositories to discover tool dependencies.

**AI responsibilities**:
- Examine build files: `go.mod`, `package.json`, `Cargo.toml`, `pyproject.toml`, `Gemfile`,
  `pom.xml`, `build.gradle`, `Makefile`, `CMakeLists.txt`
- Examine CI configs: `.github/workflows/*.yml`, `.gitlab-ci.yml`, `Jenkinsfile`
- Examine dev tool configs: `.tool-versions`, `.nvmrc`, `.python-version`, `.sdkmanrc`
- Examine Dockerfiles/Containerfiles for hints about required system packages
- Deduplicate across repos (if Go 1.23 found in two repos, list once)
- Resolve conflicts (if repo A needs Go 1.22 and repo B needs Go 1.23, suggest >= 1.23)
- Present findings to user for review before writing to manifest

**Script responsibilities**:
- YAML manipulation (reading/writing `cc-deck-build.yaml`)
- Path validation (checking repos exist)

**Interaction flow**:
```
User: /cc-deck.extract
AI:   "Which repositories should I analyze? Provide paths to local checkouts."
User: "/path/to/repo1, /path/to/repo2"
AI:   [analyzes build files, CI configs, tool configs]
AI:   "I found the following tool dependencies:
       - Go 1.23 (from repo1/go.mod)
       - Python 3.12 (from repo2/pyproject.toml)
       - protoc >= 25.0 (from repo1/.github/workflows/ci.yml)

       Shall I add these to cc-deck-build.yaml?"
User: "Yes, but skip protoc"
AI:   [updates manifest]
```

### `/cc-deck.plugin` - Plugin Management

**Purpose**: Add Claude Code plugins to the manifest.

**AI responsibilities**:
- List available plugins from configured marketplaces
- Accept direct coordinates (git URLs, marketplace names)
- Validate plugin existence
- Present current plugin list for review

**Script responsibilities**:
- Query marketplace API (if available)
- YAML manipulation

**Interaction flow**:
```
User: /cc-deck.plugin
AI:   "Current plugins in manifest: (none)
       I can add plugins from:
       1. Your configured marketplaces
       2. Direct git URL

       Which plugins would you like to add?"
User: "Add cc-rosa and sdd from marketplace"
AI:   [validates, updates manifest]
```

### `/cc-deck.mcp` - MCP Server Configuration

**Purpose**: Add MCP servers as sidecar containers.

**AI responsibilities**:
- Help user select from a catalog file (`mcp-sidecars.yml`) if present
- Read container image labels (via `podman inspect`) for auto-configuration
- Determine auth requirements from labels
- Generate appropriate compose/k8s sidecar config

**Script responsibilities**:
- `podman inspect` to read labels
- YAML manipulation

**Label schema on MCP images** (read by this command):
```
cc-deck.mcp/name: "github"
cc-deck.mcp/version: "1.0.0"
cc-deck.mcp/transport: "sse"
cc-deck.mcp/port: "8000"
cc-deck.mcp/auth-type: "token"
cc-deck.mcp/auth-env-vars: "GITHUB_TOKEN"
cc-deck.mcp/description: "GitHub MCP server"
cc-deck.mcp/docs-url: "https://..."
```

**Interaction flow**:
```
User: /cc-deck.mcp
AI:   "I'll help add MCP servers. Provide an image reference or I can check
       your mcp-sidecars.yml catalog."
User: "ghcr.io/modelcontextprotocol/github-mcp:latest"
AI:   [pulls image, reads labels]
AI:   "This is a GitHub MCP server (SSE transport, port 8000).
       It requires GITHUB_TOKEN for authentication.
       Shall I add it to the manifest?"
```

### `/cc-deck.containerfile` - Containerfile Generation

**Purpose**: Generate a complete Containerfile from the manifest. Always regenerates from scratch.

**AI responsibilities**:
- Resolve free-form tool descriptions to concrete install commands
  (e.g., "Go compiler >= 1.22" becomes `dnf install -y golang` or downloads from golang.org)
- Optimize layer ordering (rarely-changing tools first)
- Handle multi-stage builds if beneficial
- Include plugin installation steps
- Include cc-deck settings/hooks if specified in manifest
- Add a `# GENERATED BY cc-deck.containerfile - DO NOT EDIT` header

**Script responsibilities**:
- Read manifest YAML
- Write Containerfile

**Layer strategy**:
```dockerfile
# GENERATED BY cc-deck.containerfile - DO NOT EDIT
# Regenerate with: claude /cc-deck.containerfile

FROM ghcr.io/cc-deck/base:latest

# Layer: System packages (changes rarely)
RUN dnf install -y golang protobuf-compiler && dnf clean all

# Layer: Language-specific tools (changes occasionally)
RUN npm install -g pnpm@9

# Layer: Claude Code plugins (changes occasionally)
RUN claude plugins install cc-rosa sdd

# Layer: MCP client configuration
COPY mcp-config.json /home/coder/.claude/

# Layer: Project configuration (changes often)
COPY CLAUDE.md /home/coder/
COPY settings-hooks.json /home/coder/.claude/settings.json
```

### `/cc-deck.publish` - Convenience Build + Push

**Purpose**: Single command that runs `cc-deck build` then `cc-deck push`.

This is a thin wrapper that calls the CLI commands. Exists as a Claude Code command
for convenience (user can do everything from within the AI session).

## Embedding in the Go Binary

All commands and scripts are embedded using `go:embed`, following the same pattern as
the plugin WASM file and layout templates. The `cc-deck build init` command extracts
them into the build directory.

For script discovery (the spec-kit pattern referenced by the user), scripts are located
relative to the command files. The command `.md` files reference scripts by relative path.

## Open Questions

- Should `/cc-deck.extract` support re-running on already-analyzed repos?
  (Yes, it should update existing entries and flag changes.)
- How to handle tool version conflicts across repos?
  (LLM suggests the highest compatible version, user confirms.)
- Should we generate a `.dockerignore` alongside the Containerfile?
  (Yes, to exclude `.claude/` directory and other non-image files.)
