= Volume Mounts

Mount your local source code into the container so Claude Code works directly on your files.
Changes are bidirectional: edits inside the container appear on your host, and vice versa.

== Mount a Project Directory

[source,bash]
----
podman run -d --name mydev \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -v ~/projects/my-app:/home/coder/my-app \
  quay.io/cc-deck/cc-deck-demo:latest
----

Inside the container, the project is at `/home/coder/my-app`.

== Mount Multiple Projects

[source,bash]
----
podman run -d --name mydev \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -v ~/projects/frontend:/home/coder/frontend \
  -v ~/projects/backend:/home/coder/backend \
  -v ~/projects/infra:/home/coder/infra \
  quay.io/cc-deck/cc-deck-demo:latest
----

Each project gets its own Claude Code session.
The sidebar tracks them all.

== Persistent State

To preserve container state (Zellij sessions, Claude Code history, snapshots) across container restarts, mount a volume for the home directory:

[source,bash]
----
podman run -d --name mydev \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -v mydev-home:/home/coder \
  -v ~/projects/my-app:/home/coder/my-app \
  quay.io/cc-deck/cc-deck-demo:latest
----

The named volume `mydev-home` persists even when the container is removed.

WARNING: When mounting a volume on `/home/coder`, the container's default configuration files (`.zshrc`, `.config/zellij/`) are replaced by the volume contents.
On the first run, these files are populated from the image.
On subsequent runs, the volume's contents take precedence.

== File Permissions

The container runs as `coder` (UID 1000).
If your host files are owned by a different UID, you may see permission issues.

On Linux, the simplest fix is `--userns=keep-id`:

[source,bash]
----
podman run -d --name mydev \
  --userns=keep-id \
  -v ~/projects/my-app:/home/coder/my-app \
  quay.io/cc-deck/cc-deck-demo:latest
----

This maps your host UID to the container's `coder` UID.

On macOS, file permissions are handled automatically by the Podman VM.

== Read-Only Mounts

For reference material you do not want modified:

[source,bash]
----
-v ~/docs/api-spec:/home/coder/api-spec:ro
----

The `:ro` suffix makes the mount read-only inside the container.
