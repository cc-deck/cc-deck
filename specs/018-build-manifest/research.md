# Research: cc-deck Build Pipeline

**Date**: 2026-03-12
**Branch**: 018-build-manifest

## go:embed Pattern for Commands and Scripts

### Decision: Extend existing embed pattern with embed.FS for directories

**Rationale**: The current cc-deck uses `//go:embed cc_deck.wasm` for a single binary file.
For the build pipeline, we need to embed multiple files (5 commands + helper scripts).
Go's `embed.FS` allows embedding entire directories.

**Pattern**:
```go
//go:embed build-commands/*
var buildCommands embed.FS

//go:embed build-scripts/*
var buildScripts embed.FS
```

Files are extracted during `cc-deck build init` using `fs.WalkDir` to iterate and write.

**Alternatives considered**:
- Single file embeds per command (rejected: too many embed directives)
- External download at init time (rejected: offline support needed)
- Tar archive embed (rejected: unnecessary complexity)

## Manifest Schema Design

### Decision: YAML with version field, flat structure, no nesting beyond 2 levels

**Rationale**: YAML is human-readable and familiar to the target audience (developers
who work with k8s manifests, CI configs, etc.). The schema is intentionally flat to
keep it simple for manual editing.

**Version field**: `version: 1` at top level. Schema changes increment this.
No backward compatibility handling in v1 (addressed if needed when v2 arises).

**Validation**: Use `serde/yaml` in Go (via `gopkg.in/yaml.v3`) with struct tags.
Schema validation happens in `cc-deck build` before invoking the container runtime.

## Claude Code Command Format

### Decision: Markdown command files with YAML frontmatter

**Rationale**: Follow the existing `.claude/commands/*.md` pattern used by spec-kit.
Commands have a `description` in frontmatter and numbered steps in the body.

**Commands to embed**:
1. `cc-deck.extract.md` - Repository analysis for tool discovery
2. `cc-deck.plugin.md` - Plugin management
3. `cc-deck.mcp.md` - MCP server configuration
4. `cc-deck.containerfile.md` - Containerfile generation
5. `cc-deck.publish.md` - Build + push convenience wrapper

**Helper scripts** (called by commands):
- `validate-manifest.sh` - YAML schema validation
- `update-manifest.sh` - YAML section updates (avoids LLM touching raw YAML)

## Container Runtime Detection

### Decision: Auto-detect podman first, fall back to docker

**Rationale**: The project prefers podman (per CLAUDE.md). Check `command -v podman`
first, then `command -v docker`. The runtime choice is transparent to the user.

**Implementation**: A shared utility function `detectRuntime() string` used by
`build`, `push`, and `verify` commands.

## cc-deck Self-Embedding in User Images

### Decision: Copy binary to build context, Containerfile runs `cc-deck plugin install`

**Rationale**: The cc-deck binary knows its own WASM plugin version and compatible
Zellij version. By copying itself into the image and running `plugin install`, it
ensures version consistency without external coordination.

**Flow**:
1. `cc-deck build` finds its own executable via `os.Executable()`
2. Copies it to `<build-dir>/.build-context/cc-deck`
3. The Containerfile has `COPY cc-deck /usr/local/bin/cc-deck`
4. Then `RUN cc-deck plugin install --install-zellij --force`

The `.build-context/` directory is in `.gitignore` (ephemeral).

## Build Directory .gitignore

### Decision: Ignore generated files, keep manifest and commands

**Generated (ignored)**:
- `Containerfile` (regenerated by AI)
- `.build-context/` (ephemeral build artifacts)
- `compose.yaml` (generated by deploy, future spec 020)
- `k8s/` (generated by deploy, future spec 020)

**Tracked (not ignored)**:
- `cc-deck-build.yaml` (manifest, source of truth)
- `.claude/commands/` (embedded at init, but user might customize)
- `.claude/scripts/` (embedded at init)
