## Summary

Add `cargo-llvm-cov` based coverage measurement to the cc-deck WASM plugin project. This includes three Makefile targets for local use (`coverage`, `coverage-summary`, `coverage-json`), extending the CI `rust-test` job to upload lcov to Codecov with a `rust` flag, and updating the README badge to reflect Rust coverage. The existing aggregate Codecov badge already covers both Go and Rust once data is uploaded.

## Phase 0: Research

### R1: cargo-llvm-cov Usage and Output Formats

**Decision**: Use `cargo-llvm-cov` with LLVM source-based instrumentation
**Rationale**: Works on macOS and Linux, produces accurate line coverage, supports HTML/lcov/JSON output, widely adopted in the Rust ecosystem
**Alternatives considered**:
- `cargo-tarpaulin`: Linux-only, does not work on macOS (developer platform)
- `grcov`: Requires manual setup of LLVM profraw files, more complex than cargo-llvm-cov

**Key commands**:
- `cargo llvm-cov --html` - HTML report
- `cargo llvm-cov --lcov --output-path lcov.info` - lcov format for Codecov
- `cargo llvm-cov --json` - JSON format for machine processing
- `cargo llvm-cov --ignore-filename-regex 'tests?\.rs'` - exclude test files

### R2: Per-Module Summary Approach

**Decision**: Use `cargo llvm-cov --json` output and parse with `jq` to produce a per-module table
**Rationale**: The JSON output contains per-file coverage data. Grouping files by their parent directory (controller/, sidebar_plugin/, root) gives the per-module breakdown. Using `jq` avoids adding any Rust/Python dependencies.
**Alternatives considered**:
- Custom Rust script: overkill for parsing JSON
- `cargo llvm-cov report`: does not support per-module grouping natively

**Module mapping** (top-level source directories):
- `controller/` - Controller module (actions, events, hooks, state, render, sidebar_registry)
- `sidebar_plugin/` - Sidebar module (input, modes, render, rename, state)
- Root files - `main.rs`, `lib.rs`, `session.rs`, `pipe_handler.rs`, `config.rs`, `perf.rs`, `git.rs`, `debug.rs`, `wasm_compat.rs`

### R3: CI Integration Pattern

**Decision**: Mirror the existing `go-test` job pattern in `rust-test`
**Rationale**: The Go job already uses `codecov/codecov-action@v5` with `flags: go` and conditional token check. Replicating this pattern for Rust ensures consistency.
**Key changes to `rust-test` job**:
1. Add `llvm-tools-preview` component to `dtolnay/rust-toolchain@stable`
2. Install `cargo-llvm-cov` via `taiki-e/install-action@cargo-llvm-cov`
3. Replace `cargo test` with `cargo llvm-cov --lcov --output-path lcov.info`
4. Add `codecov/codecov-action@v5` step with `flags: rust`

### R4: README Badge

**Decision**: The existing aggregate Codecov badge already covers all flags. Once Rust coverage is uploaded alongside Go, the aggregate badge reflects combined coverage. No separate Rust-specific badge is needed unless the team wants flag-specific badges.
**Rationale**: Codecov aggregates all flags by default. Adding a second flag-specific badge URL would be `https://codecov.io/gh/cc-deck/cc-deck/branch/main/graph/badge.svg?flag=rust` but this adds visual clutter. The existing badge serves the purpose.
**Decision**: Keep the existing aggregate badge. Document how to add a flag-specific badge if desired later.

### R5: Makefile Error Handling

**Decision**: Check for `cargo-llvm-cov` at the start of each coverage target using `command -v cargo-llvm-cov`
**Rationale**: Clear error message with installation instructions is better than a cryptic "command not found" error from make.
