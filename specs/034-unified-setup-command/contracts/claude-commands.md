# Claude Command Contracts

## /cc-deck.capture

**Purpose**: Discover local tools, shell configuration, Claude Code plugins, and MCP servers. Write findings into the shared sections of the manifest. Target-agnostic.

### Input

`$ARGUMENTS`: Optional repository paths or section-specific flags. If empty, scans cwd and immediate children for git repositories.

### Behavior

1. **Repository analysis**: Scan build files, CI configs, version files, container files. Extract tool requirements with versions and provenance.
2. **Shell configuration**: Analyze the user's shell RC file. Curate a container/remote-ready config by stripping macOS-specific and local-path entries.
3. **Zellij configuration**: Scan `~/.config/zellij/` for user config. Exclude cc-deck managed files (layouts, plugin WASM).
4. **Claude configuration**: Scan `~/.claude/` for CLAUDE.md, settings.json. Merge strategy for settings preserves cc-deck hooks.
5. **Plugin discovery**: Read installed plugins from `~/.claude/settings.json` and plugin directories.
6. **MCP server discovery**: Read MCP configs from cc-setup cache and Claude settings.
7. **Present findings**: Show summary per section. User accepts, edits, or skips each section.
8. **Write manifest**: Update shared sections only. Never touch `targets` section.

### Sections Modified

| Section | Source |
|---------|--------|
| `tools` | Build files, CI configs, version files |
| `sources` | Repository URLs and detected tools |
| `github_tools` | GitHub release tools from CI configs |
| `settings` | Shell, Zellij, Claude configuration |
| `plugins` | Claude Code plugin discovery |
| `mcp` | MCP server configs |
| `network` | Domain groups from ecosystem files |

### Sections NOT Modified

- `targets.container` (set by init or manual edit)
- `targets.ssh` (set by init or manual edit)

### Idempotency

Re-running capture on an already-populated manifest shows current selections and allows modifications. Existing entries are updated, not duplicated.

## /cc-deck.build

**Purpose**: Generate artifacts from the manifest and apply them to the specified target.

### Input

`$ARGUMENTS`: Must include `--target container` or `--target ssh`. Optional: `--push` (container only).

### Behavior: --target container

Same as current `/cc-deck.build` behavior (see existing `cc-deck.build.md`):

1. Read and validate manifest (`cc-deck-setup.yaml`, field: `targets.container`)
2. Generate Containerfile from manifest
3. Handle existing Containerfile (show diff, ask user to choose)
4. Prepare build context (download cc-deck binaries)
5. Build image via podman (10-minute timeout, multi-arch)
6. Self-correction loop (up to 3 retries on build failures)
7. Generate `build.sh` for standalone rebuilds
8. Report results (image name, size, retries)
9. If `--push`: push to registry from `targets.container.registry`

### Behavior: --target ssh

1. Read and validate manifest (`cc-deck-setup.yaml`, field: `targets.ssh`)
2. Check that `ansible-playbook` is available locally. If not, error: "Ansible is required for SSH provisioning. Install with: brew install ansible"
3. Generate Ansible artifacts:
   - `inventory.ini` from `targets.ssh` fields
   - `group_vars/all.yml` from shared manifest sections + cc-deck/zellij versions
   - `site.yml` playbook including all roles
   - Role task files (`roles/*/tasks/main.yml`) generated from manifest content
4. Handle existing playbooks (show diff if roles already have content, ask user to choose)
5. Run `ansible-playbook -i inventory.ini site.yml` from the setup directory
6. Self-correction loop (up to 3 retries): read Ansible error output, fix the relevant role, re-run. Ansible idempotency ensures already-succeeded tasks are skipped.
7. Generate `README.md` with standalone usage instructions
8. Report results (host, roles applied, retries)

### Target Validation

| Condition | Action |
|-----------|--------|
| `--target` not provided | Error: "Specify --target container or --target ssh" |
| Target section missing from manifest | Error: "No [target] section in manifest. Run cc-deck setup init --target [target] or add it manually." |
| `--push` with `--target ssh` | Error: "--push is only valid with --target container" |

### Self-Correction Contract

Both targets share the same correction pattern:
1. Run the build/apply command
2. If failure, read error output
3. Identify failing step and root cause
4. Fix the relevant artifact (Containerfile or role task file)
5. Retry (cached/idempotent work is skipped)
6. After 3 failed attempts, stop and report the error with analysis

### Generated Artifacts Ownership

All generated files include a "GENERATED BY cc-deck.build - DO NOT EDIT MANUALLY" header. The manifest is the source of truth. Regeneration overwrites generated files (with user confirmation if content differs).
