# Implementation Plan: Render Pipeline Stability and CPU Optimization

**Branch**: `053-render-pipeline-stability` | **Date**: 2026-05-14 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/053-render-pipeline-stability/spec.md`

## Summary

Fix render pipeline instability (flickering, blinking, CPU waste) by eliminating the phantom second controller instance, adding push-on-discovery for sidebar bootstrapping, making render triggers conditional, and adding profiling instrumentation. The primary approach is root-cause investigation of the dual controller with a defensive plugin_id guard as the fallback.

## Technical Context

**Language/Version**: Rust stable, edition 2021 (wasm32-wasip1 target)
**Primary Dependencies**: zellij-tile 0.44, serde/serde_json 1.x
**Storage**: WASI `/cache/` directory for debug logs, perf data, session state
**Testing**: `cargo test` (unit), manual verification with debug logging and perf CSV
**Target Platform**: WASM (wasm32-wasip1), executed by Zellij's wasmi interpreter
**Project Type**: Zellij plugin (controller + sidebar in single WASM binary)
**Performance Goals**: Under 30% CPU with 14 idle tabs, zero renders/sec at steady state
**Constraints**: No async I/O in WASI, all plugin instances share `/cache/`, single-threaded execution per instance
**Scale/Scope**: 14+ concurrent plugin instances (1 controller + N sidebars)

## Implementation Phases

### Phase 1: Defensive Guard and Investigation (FR-007, FR-001, FR-002)

The highest-impact, lowest-risk change. Guards the controller against processing messages when plugin_id is unset (permissions not granted). This is the defensive fallback that works regardless of root cause.

**Changes**:
1. `controller/state.rs`: Add `disabled: bool` field to ControllerState
2. `controller/mod.rs`: In `pipe()` handler, skip all processing if `!state.permissions_granted` (plugin_id == 0). In `update()` handler for Timer, skip if disabled.
3. `pipe_handler.rs`: Add `ControllerPing`, `ControllerPong` variants
4. `debug.rs`: Add plugin_id prefix to log lines for disambiguation

**Tests**:
- Unit test: controller with permissions_granted=false ignores pipe messages
- Unit test: controller with disabled=true ignores timer events

### Phase 2: Startup Probe Protocol (FR-005, FR-006)

Implement the controller-ping/pong protocol for detecting and resolving dual controllers. The controller with the higher plugin_id self-disables.

**Changes**:
1. `controller/mod.rs`: On PermissionRequestResult, broadcast `cc-deck:controller-ping` with own plugin_id as payload
2. `controller/mod.rs`: Handle incoming `cc-deck:controller-ping`: if sender has lower plugin_id, respond with pong and self-disable. If sender has higher plugin_id, send pong (sender will self-disable).
3. `controller/mod.rs`: Handle incoming `cc-deck:controller-pong`: if responder has lower plugin_id, self-disable.
4. `controller/state.rs`: Set `disabled = true` on self-disable, log warning

**Tests**:
- Unit test: lower plugin_id wins when two controllers exist
- Unit test: disabled controller does not process any messages

### Phase 3: Sidebar Bootstrapping (FR-008, FR-009, FR-010)

Push-on-discovery for immediate render, plus one-shot fallback request.

**Changes**:
1. `controller/sidebar_registry.rs`: `discover_sidebars_from_manifest` returns Vec<u32> of newly registered plugin_ids
2. `controller/events.rs`: After discover call, send targeted render to each new sidebar
3. `sidebar_plugin/state.rs`: Add `render_request_sent: bool` and `ticks_since_init: u8`
4. `sidebar_plugin/mod.rs`: Subscribe to Timer events. On tick, increment ticks_since_init. If ticks_since_init >= 3 and !initialized and !render_request_sent, send `cc-deck:render-request` (only if controller_plugin_id is known from SidebarInit) and set render_request_sent = true. After first render payload is received (initialized = true), unsubscribe from Timer to avoid per-second update() overhead. Guard the render-request against missing controller_plugin_id: if SidebarInit has not arrived yet, wait for the next tick rather than sending to an unknown target
5. `pipe_handler.rs`: Add `RenderRequest` variant
6. `controller/mod.rs`: Handle `RenderRequest` by sending targeted render to the requesting sidebar
7. `sidebar_plugin/render.rs`: Display "Connecting..." when cached_payload is None

**Tests**:
- Unit test: discover_sidebars_from_manifest returns new IDs
- Unit test: sidebar sends render request after 3 ticks with no payload
- Unit test: sidebar does not send render request if payload already received
- Unit test: sidebar does not send render request more than once

### Phase 4: Conditional Render Triggers (FR-011, FR-012, FR-013)

Remove unconditional mark_render_dirty from handle_tab_update.

**Changes**:
1. `controller/events.rs`: Replace unconditional `state.mark_render_dirty()` at end of handle_tab_update with conditional logic:
   - Mark dirty if active tab changed (new_active != previous active_tab_index)
   - Mark dirty if tab_count_changed
   - Mark dirty if remove_dead_sessions returned true
   - Mark dirty if cleanup_stale_sessions returned true
   - Do NOT mark dirty otherwise

**Tests**:
- Unit test: handle_tab_update with no changes does not mark dirty
- Unit test: handle_tab_update with tab count change marks dirty
- Unit test: handle_tab_update with active tab change marks dirty

### Phase 5: Debug and Profiling Improvements (FR-003, FR-004)

Optimize debug logging and add render profiling.

**Changes**:
1. `debug.rs`: Add buffered logging:
   - Module-level `LOG_BUFFER: Vec<String>` (behind `RefCell` in a thread_local, avoiding `static mut` and `unsafe`)
   - `debug_log()` pushes to buffer instead of writing
   - `debug_flush()` writes all buffered lines and clears buffer
   - `debug_flush()` called from timer handler and when buffer exceeds 50 lines
   - Note: Use `thread_local! { static LOG_BUFFER: RefCell<Vec<String>> }` to avoid unsafe. WASI is single-threaded so thread_local has no overhead.
2. `controller/render_broadcast.rs`: In `broadcast_render`, wrap serialization and pipe sends with perf timing:
   - Record `render:broadcast` with serialization time
   - Record `render:pipe_send` count (one per sidebar)
3. `controller/render_broadcast.rs`: In `flush_render`, record `render:skipped` when not dirty

**Tests**:
- Unit test: debug_log buffers, debug_flush writes and clears
- Unit test: perf events recorded during broadcast

### Phase 6: Integration Testing and Verification

Validate all changes together against success criteria.

**Changes**:
